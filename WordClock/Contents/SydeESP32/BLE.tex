\chapter{BLE Connection Check}
Bluetooth Low Energy (BLE) is an energy-efficient communication method widely used in embedded systems for wireless data transmission. The Hypatia Board features a BLE interface, which will be evaluated for its functionality in the following section. This includes identifying the BLE module, initializing the connection, and analyzing signal quality and data transmission. The objective of this investigation is to validate the stability and reliability of BLE communication of the Hypatia Board and identify potential sources of error. 

\section {Explanations}
\subsection{UUID}
A UUID (Universally Unique Identifier) is a 128-bit unique identifier used in various applications, including Bluetooth Low Energy (BLE), databases, and distributed computing. In BLE communication, UUIDs play a crucial role in identifying specific services, characteristics, and descriptors, ensuring that different devices and applications do not interfere with each other.  

In BLE, UUIDs can be categorized into two main types: standard 16-bit UUIDs and custom 128-bit UUIDs. Standard UUIDs, assigned by the Bluetooth Special Interest Group (SIG), are used for predefined services such as the Battery Service (0x180F) or the Heart Rate Service (0x180D). These identifiers are commonly used in BLE devices to provide standardized functionality. In contrast, custom 128-bit UUIDs such as here needed in the purpose of getting the Hypatia Booard started, allow developers to define unique identifiers for proprietary services and characteristics, ensuring compatibility within custom BLE applications. An example of a custom UUID would be `"b0cee666-6f5c-48fa-9b6b-ee42fec51a13"`, which could be used to identify a specific BLE service running on an ESP32.  

The primary advantage of UUIDs is their ability to prevent conflicts in environments where multiple BLE devices are operating simultaneously. By assigning a unique UUID to a service or characteristic, developers can ensure that only the intended client device connects and interacts with the correct BLE service. This is particularly useful in IoT applications, where multiple smart devices communicate wirelessly. UUIDs are automatically generated using algorithms which can be found online, that minimize the probability of duplication, making them a reliable and scalable solution for device identification.

At the beginning of the code, the necessary libraries are included, and constants are defined to specify the LED pin (GPIO 21) and the number of LEDs (NUM\_LEDS 1). The UUIDs for the BLE service and characteristic are also defined, ensuring unique identification within the BLE environment. A NeoPixel object is then created to control the LED, allowing it to display different colors.

To process incoming BLE commands, a custom callback class (MyCallbacks) is implemented. When a new value is written to the characteristic, the function onWrite() extracts the received data. If the message contains "{}on{}", the LED turns blue, while if "{}off{}" is received, the LED turns off. The received data is also printed to the serial monitor for debugging purposes.

The setup() function initializes the ESP32’s BLE server and configures the LED. Serial communication is started at 115200 baud, and the LED is initially turned off. The ESP32 is then set up as a BLE device named "MyESP32", and a BLE server is created. Within this server, a BLE service and characteristic are defined, allowing the characteristic to be read and written. A callback function is assigned to handle incoming commands, and an initial value ("Hello World") is set.

Once the BLE service is started, the ESP32 begins advertising the service UUID, making it discoverable by external devices. This enables a BLE scanner, such as a smartphone app, to detect and connect to the ESP32. The loop() function contains only a small delay, as the BLE functionality operates asynchronously in the background.

When the ESP32 is powered on, it starts advertising itself as "MyESP32". A client device, such as a smartphone running a BLE scanning app, can discover and connect to it. Once connected, the client can send "{}on{}" or "{}off{}" commands via BLE. If "{}on{}" is received, the LED changes to blue, while "{}off{}" turns the LED off. The ESP32 processes these commands in real-time and updates the LED accordingly.

\section{Connecting Hypatia with BLE Scanner}

To proceed with the setup and programming of the Hypatia Board, it is essential to install the Arduino IDE on your computer. Additionally, the Adafruit Library is required for proper functionality and communication with specific components of the Hypatia Board. Make sure that all necessary configurations for the Hypatia Board have been properly set up. When the Configurations are set and the correct Board and COM Port are chosen, its time to compile and upload the explained code to the ESP32 on the Hypatia board by pressing the arrow button in the upper left corner of the Arduino IDE. The next step is to open the Serial Monitor Tab via the button in the upper right corner like shown in the following figure \ref{fig:ArduinoSerialMonitor}.

Choose the baudrate in the upper right corner of the Serial monitor tab which was initialized in the code (115200 baud). This allows you to monitor real-time feedback from the board (see Figure \ref{fig:BLESErialMonitor}), ensuring that the initialization, connection, and communication processes are executed correctly. If the Serial Monitor is not showing text, try to upload the code again with the serial monitor being open.
